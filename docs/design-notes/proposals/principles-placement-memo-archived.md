# 設計メモ: PRINCIPLES.md（設計原則ドキュメント）

> ステータス: 配置方針確定・内容未策定
> 着手予定: 任意のタイミング

---

## 目的

Aphelion の設計原則を定め、Aphelion 自身の開発が思想から外れないようにする。
**基本的に変わることのない（揺るがない）思想** を記述する。
実装詳細やエージェント構成（変わりうるもの）は含めない。

---

## 配置方針

### 配置場所: リポジトリルート直下

```
PRINCIPLES.md  ← ルート直下（README.md / CHANGELOG.md / CONTRIBUTING.md と並ぶ）
```

### `.claude/rules/` への配置を避ける理由

- 全39エージェントへの自動ロードが発生しトークンコストが増加する
- token-reduction-memo / agent-definition-simplification-memo の削減方針と逆行する

### 参照方式: 手動ロード

- レビュー時にエージェントへ手動で PRINCIPLES.md をロードして思想準拠を確認する
- 自動ロードしないことでトークンコストを制御する

---

## 補完策（手動ロード忘却リスクへの対応）

1. **CONTRIBUTING.md に明記**（記載予定）
   - レビュー前に PRINCIPLES.md をロードすることをルール化
2. **PRテンプレートにチェックボックス追加**
   - 「PRINCIPLES.md に照らして確認済み」
3. **`/aphelion-check` に存在確認を追加**（setup-improvement-memo と連動）

---

## doc-reviewer との役割分離

- doc-reviewer: ドキュメント整合性チェック（SPEC.md 中心）
- 思想準拠チェック: 人間が承認ゲート（HITL）で担う役割
- doc-reviewer の責任範囲からは「思想準拠チェック」を明示的に除外する

---

## 内容の方針（策定時の指針）

- **揺るがない原則のみ記述** — エージェント数・フロー構成など変わりうる実装詳細は含めない
- 原則レベルは変わりにくい抽象度で記述する
- 想定される原則の例（策定時に精査）:
  - HITL ゲートを各フェーズ境界に置く（人間の承認なしに次へ進まない）
  - 1エージェント1責任
  - ドメイン間は .md ファイルでハンドオフし、自動チェーンしない
  - トリアージにより不要な処理をスキップする
  - Claude Code を正準ソースとする
  - 破壊的操作には監査証跡を残す
